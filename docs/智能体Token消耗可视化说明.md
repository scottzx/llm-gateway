# 智能体 Token 消耗可视化 - 使用指南

## 核心概念：为什么会有重复消耗？

大模型本身是**无状态**的，它没有记忆。智能体需要为每一轮对话重新构建完整的上下文，把所有必要的信息再次发送给大模型。这就是为什么每一轮都会产生 token 消耗。

> **关键点**：即使是同样的工具定义或系统提示，每一轮对话都需要重新发送，这就是为什么 Tools-Reminder 会占据如此大的比例。

---

## 数据解读

### 总体统计（左侧卡片）

- **总计**：整个对话过程的总消耗
- **平均**：每轮对话的平均消耗
- **峰值 vs 谷值**：展示了不同轮次的巨大差异

### 角色分布（重点！）

> **Tools-Reminder 通常占据最大比例（可能达到 60-80%）**

#### 原因：

1. **工具定义很详细**：每个工具都有完整的 JSON schema，包含参数类型、描述、验证规则等
2. **每轮都发送**：只要启用了工具，每一轮请求都要携带完整的工具定义
3. **累积效应**：从第 1 轮到第 N 轮，工具定义被重复发送了 N 次

#### 其他角色说明：

| 角色 | 说明 |
|------|------|
| **System-Reminder** | 动态注入的系统提示 |
| **System** | 基础系统提示词 |
| **Tool** | 实际的工具调用和结果 |
| **Assistant** | 模型的回复内容 |
| **User** | 用户的实际输入（通常占比很小） |

### 每轮对话柱状图

每一根柱子代表一轮对话的 token 消耗：

- **详细数据**：鼠标悬停在柱子上可以看到该轮的详细分解
- **波动模式**：不同轮次的差异很大
  - 工具调用多的轮次消耗就高
  - 简单对话的轮次消耗就低

#### 颜色编码

- 灰色系：系统级内容（System、System-Reminder、Tools-Reminder）
- 蓝色：用户输入（User）
- 绿色：助手回复（Assistant）
- 紫色：工具调用（Tool）

---

## 关键洞察

### 1. 工具定义是主要成本

如果你的智能体使用了大量工具，工具定义的重复发送会成为主要的 token 消耗来源。

### 2. 输入远大于输出

通常输入/输出比例约 **6:1**。

说明大部分 token 消耗在"告诉模型做什么"，而不是"模型生成答案"。

### 3. 优化方向

- **工具精简**：只启用必要的工具
- **工具描述优化**：简洁的描述可以减少 tokens
- **上下文管理**：智能地过滤和压缩历史对话

---

## 技术说明

### 数据来源

- ✅ **准确值**：从 API 响应的 `usage` 字段提取
- 📐 **估算值**：基于文本长度（中文 2 字符/token，英文 4 字符/token）

### 角色分类

| 角色 | 说明 |
|------|------|
| **System** | 系统提示词 |
| **System-Reminder** | `<system-reminder>` 标签内容 |
| **Tools-Reminder** | `tools` 数组的 JSON schema |
| **User** | 用户实际输入 |
| **Assistant** | 助手回复文本 |
| **Tool** | 工具调用和结果 |

---

## 总结

在使用工具的智能体场景中，**工具定义的重复发送是最大的 token 成本来源**。

理解这一点，对于优化智能体的成本和性能至关重要。

---

## 示例数据解读

假设一个 31 轮的对话：

```
总计: 334K tokens
平均: 3.9K tokens/轮
峰值: 32.1K tokens
谷值: 12 tokens

角色分布:
- Tools-Reminder: 207K (62%) ← 最大消耗
- System-Reminder: 37K (11%)
- System: 23K (7%)
- Tool: 27K (8%)
- Assistant: 7K (2%)
- User: <1K (<1%)
```

从这个数据可以看出：
- 工具定义被重复发送了 31 次
- 用户实际输入只占不到 1%
- 大部分 token 消耗在系统级内容上
